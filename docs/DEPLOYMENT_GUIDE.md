# Fabric IQ — Customer Tenant Deployment Guide

> **Short, table-driven runbook.** Manual steps create only the three Azure resources that require human decisions (Resource Group, Fabric capacity, Foundry account). Everything else is provisioned, wired, and verified by five idempotent PowerShell scripts under [scripts/deploy/](../scripts/deploy/).

**Solution short name:** `fabriciq` (Fabric IQ — Parts Shortages)
**Single source of truth:** [deployments/config/azure.config.json](../deployments/config/azure.config.json) — every script and workflow reads from this file.

## 1. Prerequisites

### 1.1 Required Access (Who Does What)

Most steps can be performed by a single **Subscription Owner**. Two items must be handled by people outside the Azure team — call them out early because they may sit in a different group with their own change-control process.

| Who | Required role | What it enables | How long |
|---|---|---|---|
| **You / Azure platform team** | **`Owner` on the target Azure subscription** | Creates the resource group, Fabric capacity, Foundry account, App Services, Cosmos DB, VNet, all role assignments. Subscription Owner already includes `Contributor` + `User Access Administrator`, so no extra roles are required at the Azure layer. | One-time |
| **Microsoft Entra ID admin** (often a *separate* identity / IT team) | **`Application Administrator`** or **`Cloud Application Administrator`** on the target tenant | One-time run of [scripts/bootstrap_entra_auth.ps1](../scripts/bootstrap_entra_auth.ps1) to create the API + UI SPA app registrations, OAuth scopes, redirect URIs, and grant tenant admin consent. *Subscription Owner is not enough — Entra app registration is a tenant-level operation.* See [§1.1 a](#11a-handoff-to-the-entra-team) for what to send them. | One-time, ~10 min |
| **Microsoft Fabric tenant admin** (often the same person as the Entra admin) | **`Fabric Admin`** or **`Power BI Admin`** | One-time toggle of **"Service principals can use Fabric APIs"** = ON at [Fabric admin portal → Tenant settings → Developer settings](https://app.fabric.microsoft.com/admin-portal/tenantSettings). *Without this the deploy identity cannot create Fabric items, workspaces, or run notebooks.* | One-time, ~2 min |
| **GitHub org admin** | `admin` on the forked repo | OIDC federated-credential creation + repo-secret push (only needed when you deploy via GitHub Actions; the `-Local` path bypasses this) | One-time |

> If the same person holds Subscription Owner + Application Administrator + Fabric Admin, they can run the full pipeline end-to-end in ~30 min. In siloed enterprise tenants, expect the Entra and Fabric items to be a separate ticket with their own SLA.

#### 1.1a Handoff to the Entra Team

The Fabric IQ GitHub repo is **private**, so the identity / Entra team typically does *not* have a clone or even GitHub access. Send them a self-contained handoff bundle instead — generated by [scripts/package_entra_handoff.ps1](../scripts/package_entra_handoff.ps1).

**Automated step (run once, on the Azure team's workstation):**

```powershell
# Reads deployments/config/azure.config.json, produces a zip in out/
pwsh -File scripts/package_entra_handoff.ps1
# -> out/entra-handoff-<yyyyMMdd-HHmm>.zip
```

**What the zip contains (~13 KB, 4 files):**

| File | Purpose |
|---|---|
| `bootstrap_entra_auth.ps1` | The provisioning script (identical to the one in the repo) |
| `azure.config.handoff.json` | Sanitized config: `tenantId`, App Service names + public URL, `entra` block, expected redirect / post-logout URIs. **No subscription, Cosmos, Fabric, or Foundry IDs.** |
| `HANDOFF_README.md` | Step-by-step instructions, prereqs, exact commands, and the two GUIDs they must return |
| `SECURITY.md` | What the script does + does NOT do, what roles it needs, how to audit, how to revert |

Email / upload the zip to the Entra team along with this short ticket template:

> **Subject:** Fabric IQ — create two Entra ID app registrations on tenant `<TENANT_ID>`
>
> Attached: `entra-handoff-<timestamp>.zip`. Please unzip and follow `HANDOFF_README.md`. Estimated effort ≈ 10 min. Required role: `Application Administrator` on the target tenant.
>
> When done, please reply with the two GUIDs the script prints:
>
> - `AUTH_ENTRA_API_CLIENT_ID = <guid>`
> - `AUTH_ENTRA_UI_CLIENT_ID  = <guid>`
>
> No other access is required from your team — the Azure team will store the GUIDs as GitHub repo secrets and grant admin consent on the UI app at portal close-out.

**Manual steps the Entra team will follow** (taken directly from `HANDOFF_README.md`):

1. `az login --tenant <TENANT_ID> --allow-no-subscriptions`
2. `pwsh -File .\bootstrap_entra_auth.ps1 -AzureConfigPath .\azure.config.handoff.json -SkipSecretUpload`
3. Capture the two GUIDs from the **Summary** block.
4. Entra portal → Enterprise applications → *UI SPA app* → Permissions → **Grant admin consent**.
5. Reply on the ticket with the two GUIDs.

The Azure team then stores the GUIDs as repo secrets (this happens automatically in [scripts/deploy/02_bootstrap_identity.ps1](../scripts/deploy/02_bootstrap_identity.ps1) — pass `-EntraApiClientId` and `-EntraUiClientId` from the ticket reply).

### 1.2 Deployment Workstation (Recommended VM)

> **Recommended for any customer deployment.** A jumpbox VM in the same Azure tenant + VNet avoids the most common failure modes: GitHub-hosted-runner egress restrictions, tenant audit requirements, and "the API became unreachable after lockdown PASS 2 (§3 script 04)" because public access is disabled.

| Spec | Value | Why |
|---|---|---|
| SKU | `Standard_D2s_v5` (2 vCPU / 8 GB) | Enough for `npm ci`, `pip install`, and `pwsh` script execution |
| OS | Ubuntu 22.04 LTS *or* Windows Server 2022 | Both shells supported; Ubuntu uses ~2 GB less RAM |
| Network | **Inside the same VNet** as the API web app, in the App Service integration subnet's peer | After lockdown PASS 2, the API is only reachable from the VNet — the workstation needs to live there to keep deploying |
| RBAC | Subscription Owner (per §1.1) on the same subscription | Identical to local-workstation requirements |
| Tools | See [§1.3](#13-required-tools) | Install once |

Provisioning shortcut (run from a workstation that already has `az`):

```powershell
$RG = "rg-fabriciq-shortages"
$VNET = "vnet-fabriciq"             # the VNet created by script 01
$SUBNET = "snet-jumpbox"            # add a /28 subnet to host the VM
az network vnet subnet create -g $RG --vnet-name $VNET -n $SUBNET --address-prefix 10.10.4.0/28
az vm create -g $RG -n vm-fabriciq-deploy `
  --image Ubuntu2204 --size Standard_D2s_v5 `
  --vnet-name $VNET --subnet $SUBNET --public-ip-address "" `
  --admin-username azureuser --generate-ssh-keys `
  --assign-identity                 # so the VM's system MI can be granted Owner on the RG
```

Then SSH (via Bastion or VPN) and clone the repo to `/home/azureuser/Fabric-IQ-Ontology-Parts-Shortages` before running any deployment scripts.

**Quota items to validate ahead of time** (the three most common blockers):

1. **Fabric capacity SKU** — F2 is sometimes blocked in EA / CSP tenants; request via the [Fabric admin portal](https://app.fabric.microsoft.com/admin-portal/capacities).
2. **Azure OpenAI `gpt-4.1` TPM** — default tenant quota is 30K; the POC needs ≥50K to avoid throttling under concurrent chat. Raise via the **Quotas** blade in the Foundry portal.
3. **App Service plan B3** in the chosen region — verify with `az appservice list-locations --sku B3 -o table`.

**Estimated steady-state POC cost** (East US, ~$USD / month, list price): Fabric F2 ≈ $263, App Service B3 (1 plan, 2 apps) ≈ $110, Cosmos serverless ≈ $25, Foundry + `gpt-4.1` (1M tokens/day) ≈ $90, Key Vault + networking ≈ $15. **Total ≈ $500/mo.** Production sizing (F4 + P1v3 + provisioned throughput Cosmos) is ~3× higher.

#### 2.1a Configuration-driven naming (no hardcoded resource names)

Every resource name (App Service, VNet, subnet, private endpoint, Cosmos account, UAMI, Entra app, etc.) lives **exclusively** in [deployments/config/azure.config.json](../deployments/config/azure.config.json). PowerShell scripts under [`scripts/`](../scripts/) and [`scripts/deploy/`](../scripts/deploy/) read from this file via a small loader block (`Get-Content $AzureConfigPath | ConvertFrom-Json`) and validate required fields up-front.

**Rules for repo contributors:**

- **Do not** hardcode App Service names, resource group names, VNet/subnet names, Cosmos account names, Entra display names, or any other customer-specific identifier in a PowerShell script. Use `$azCfg.<path>` instead.
- Caller-supplied `-Param` values always win over the config file (so customers can override per-invocation without editing JSON).
- When you add a new resource category (e.g. a new private endpoint), add it to the JSON schema in both [azure.config.json](../deployments/config/azure.config.json) and [azure.config.template.json](../deployments/config/azure.config.template.json), then surface it via a `$Param = $azCfg.path.to.it` line in the script's config loader.
- New required fields must be added to the `foreach ($required in @(...))` validation block at the top of each script so a missing value fails fast with a clear error.

This is what lets the *same* PowerShell scripts run unchanged against a developer sandbox, a customer staging tenant, and a customer production tenant — only the JSON differs.

**Field reference for the most common overrides:**

| Resource type | Config path |
|---|---|
| API App Service name | `apps.api.name` |
| UI App Service name | `apps.ui.name` |
| App Service plan name | `plan.name` |
| VNet name | `networking.vnet.name` |
| App Service integration subnet | `networking.subnets.appservice.name` |
| Private endpoint subnet | `networking.subnets.privateEndpoints.name` |
| API private endpoint name | `networking.privateEndpoints.api` |
| Foundry private endpoint name | `networking.privateEndpoints.foundry` |

### Happy-path one-liner

```powershell
pwsh -File scripts/deploy/01_provision_infra.ps1
pwsh -File scripts/deploy/02_bootstrap_identity.ps1
pwsh -File scripts/deploy/03_deploy_application.ps1
pwsh -File scripts/deploy/04_lockdown_network.ps1               # PASS 1 only — verify UI proxy works
pwsh -File scripts/deploy/04_lockdown_network.ps1 -ApplyDeny    # PASS 2 — disable public API
pwsh -File scripts/deploy/05_verify_deployment.ps1 -ExpectPrivate
```

> Total wall time: ~25 min for scripts + ~30 min for workflow 09 (Fabric items + data load + ML training + deploys). Plan one half-day for a first-time deployment.

### Re-running and rollback

- Every script is idempotent — re-running is safe and is the recommended way to recover from transient failures.
- To restore public API access: `az webapp update -g <rg> -n <api> --set publicNetworkAccess=Enabled`.
- To wipe a failed deployment: delete the resource group (`az group delete`); the scripts will recreate everything next time.

## Appendix A — Manual Kudu Deploy Fallback

Use when GitHub Actions is unavailable (billing issue, restricted egress, audit hold). Script 03's `-Local` path automates these steps; this appendix documents the equivalent manual commands.

### A.1 API (Python) — vendored deps + raw Kudu zipdeploy

```powershell
$APIAPP = (Get-Content deployments\config\azure.config.json | ConvertFrom-Json).apps.api.name

# 1. Stage + vendor dependencies (CP312 manylinux_2_28 wheels for App Service Python|3.12 compatibility)
Copy-Item -Recurse -Force api\* api-stage\
python -m pip install --upgrade pip wheel
python -m pip install `
  --target api-stage\.python_packages\lib\site-packages `
  --no-compile `
  --platform manylinux_2_28_x86_64 --platform manylinux2014_x86_64 --platform any `
  --python-version 3.12 --implementation cp --abi cp312 --only-binary=:all: `
  -r api-stage\requirements.txt

# 2. POSIX-path zip (do NOT use Compress-Archive — bakes Windows backslashes; Linux unzip silently flattens)
Add-Type -AssemblyName System.IO.Compression.FileSystem
$root = (Resolve-Path .\api-stage).Path
$fs   = [System.IO.File]::Create('api-dist\api-package.zip')
$zip  = New-Object System.IO.Compression.ZipArchive($fs, [System.IO.Compression.ZipArchiveMode]::Create)
try {
  Get-ChildItem $root -Recurse -File | ForEach-Object {
    $rel = $_.FullName.Substring($root.Length + 1) -replace '\\','/'
    $e   = $zip.CreateEntry($rel, [System.IO.Compression.CompressionLevel]::Fastest)
    $es  = $e.Open(); $fr = [System.IO.File]::OpenRead($_.FullName); $fr.CopyTo($es); $fr.Dispose(); $es.Dispose()
  }
} finally { $zip.Dispose(); $fs.Dispose() }

# 3. Raw Kudu async zipdeploy + clean=true (mandatory — stale wwwroot shadows vendored deps)
$TOKEN = az account get-access-token --resource https://management.azure.com --query accessToken -o tsv
curl.exe -X POST `
  -H "Authorization: Bearer $TOKEN" `
  -H "Content-Type: application/zip" `
  --data-binary "@api-dist\api-package.zip" `
  --max-time 300 `
  "https://$APIAPP.scm.azurewebsites.net/api/zipdeploy?isAsync=true&clean=true"

# 4. Poll until status==4
Start-Sleep 30
curl.exe -H "Authorization: Bearer $TOKEN" "https://$APIAPP.scm.azurewebsites.net/api/deployments?`$top=1"
```

### A.2 UI (Angular + Express) — POSIX zip + Kudu zipdeploy

```powershell
$UIAPP = (Get-Content deployments\config\azure.config.json | ConvertFrom-Json).apps.ui.name

# 1. Build Angular SPA
npm ci
npm run build -- --configuration production

# 2. Stage server.js + node_modules + dist
$STAGE = "ui-dist\staging"
Remove-Item $STAGE -Recurse -Force -ErrorAction SilentlyContinue
New-Item -ItemType Directory $STAGE\browser -Force | Out-Null
Copy-Item dist\fabric-iq-ontology-parts-shortages\browser\* $STAGE\browser\ -Recurse
Copy-Item server.js $STAGE\
@'
{ "name":"fabric-iq-ui-host","version":"1.0.0","private":true,"main":"server.js",
  "scripts":{"start":"node server.js"},"engines":{"node":">=20"},
  "dependencies":{"express":"^4.21.2","http-proxy-middleware":"^3.0.3"} }
'@ | Set-Content $STAGE\package.json
Push-Location $STAGE; npm install --omit=dev --no-audit --no-fund --silent; Pop-Location

# 3. POSIX-path zip (Python zipfile — handles the full node_modules tree reliably;
#    the previous PowerShell System.IO.Compression loop silently truncated at ~18K
#    entries and produced 0-byte / unreadable zips on the ~24K-file UI stage)
python scripts\_zip_ui_posix.py   # outputs ui-dist\fabriciq-ui.zip

# 4. Deploy
$TOKEN = az account get-access-token --resource https://management.azure.com --query accessToken -o tsv
curl.exe -X POST -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/zip" `
  --data-binary "@ui-dist\fabriciq-ui.zip" --max-time 300 `
  "https://$UIAPP.scm.azurewebsites.net/api/zipdeploy?isAsync=true&clean=true"
```

### A.3 Single-file hot-fix via Kudu VFS

```powershell
$APIAPP = (Get-Content deployments\config\azure.config.json | ConvertFrom-Json).apps.api.name
$TOKEN  = az account get-access-token --resource https://management.core.windows.net/ --query accessToken -o tsv
Invoke-RestMethod -Method Put `
  -Uri "https://$APIAPP.scm.azurewebsites.net/api/vfs/site/wwwroot/main.py" `
  -Headers @{ Authorization = "Bearer $TOKEN"; "If-Match" = "*" } `
```

## Appendix B — GitHub Actions Billing

- GitHub Actions requires active billing on the customer org. If billing fails, workflow runs queue indefinitely with no clear error in the UI. Confirm at `https://github.com/organizations/<org>/billing` before first deploy.
- Free public-repo minutes do **not** apply if the repo is private and the org's spending limit is `$0`.
- During a billing outage: use `scripts/deploy/03_deploy_application.ps1 -Local` (uses Appendix A under the hood). Re-run the workflows after billing is restored to capture the canonical history.
- Deploy workflows pin `concurrency: 1` per branch. Do not push back-to-back commits to `main` during a deploy.

## Appendix C — Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `az login` returns `AADSTS50020` | Account is not a member of the customer tenant | Use `--use-device-code` and an external user; or guest the deploy account with the right roles |
| `gh secret set` returns 403 | `gh` token missing `admin:repo_hook` scope | `gh auth refresh -s admin:repo_hook,repo,workflow` |
| Workflow 04 fails with `GLIBC_2.33 not found` | CI runner pulled `manylinux_2_34` wheels (Ubuntu 24.04 default) | Confirm workflow YAML still pins `--platform manylinux_2_28_x86_64` — do not edit |
| Workflow 05 deploys but UI 404s on every route | `index.html` placed flat at wwwroot root (Windows zip bug) | Confirm `scripts\_zip_ui_posix.py` (POSIX paths via Python `zipfile`) was used, not `Compress-Archive` or PowerShell `System.IO.Compression` |
| UI Admin → Cosmos DB Tables shows 0 rows for every container | API web app cannot reach Cosmos: either VNet integration / DNS missing, or Cosmos account has `publicNetworkAccess=Disabled` with no private endpoint resolving to the integration VNet | Re-run `scripts/deploy/04_lockdown_network.ps1` PASS 1 (idempotent). Verify from Kudu SSH: `getent hosts <cosmos-account>.documents.azure.com` must return a `10.x` private IP; `curl -sS -o /dev/null -w "%{http_code}\n" https://<cosmos-account>.documents.azure.com:443/` must return `401` (TLS through PE OK, auth header expected). If DNS still returns a public IP, confirm the `privatelink.documents.azure.com` zone is VNet-linked and has A records for both `<account>` and `<account>-<region>`. |
| API responds 200 on `/health` but 500 on protected routes after lockdown | API web app MI never got Cosmos role | Re-run `scripts/deploy/04_lockdown_network.ps1` (PASS 1 step 5 is idempotent) |
| Foundry agent calls return `403 InvalidApiKey` | Runtime MI missing `Cognitive Services OpenAI User` on project scope | Re-run `scripts/deploy/02_bootstrap_identity.ps1` |
| `az webapp deploy --async true` hangs ~11 min then false-fails | Az CLI does a sync warmup probe even in async mode | Use the raw Kudu `curl` recipe in Appendix A.1 (returns 202 in <60s) |
| Fabric workflow 03 cannot reach workspace | "Service principals can use Fabric APIs" disabled on tenant | Fabric tenant admin enables it; or downgrades workflow 03 to PAT-based auth |
| UI redirects to `/login` after Entra success | `API_PRIVATE_FQDN` secret empty / placeholder | `gh secret set API_PRIVATE_FQDN --body "https://<APIAPP>.privatelink.azurewebsites.net"`, then re-deploy UI |
| `WEBSITES_CONTAINER_START_TIME_LIMIT` exceeded — `SiteStartupCancelled` | Container cold-start > 230s during heavy vendored-deps extraction | Already set to 600s by script 01; if still failing, check `init_container.sh` logs in Kudu |
| Existing Cosmos container has wrong partition key after upgrade (e.g. `users` was `/id`, now `/email`) | Cosmos does not allow changing a container PK in place | Re-run `scripts/deploy/01_provision_infra.ps1 -RecreateMismatchedContainers` (DROPS and recreates the container — all docs lost). Re-seed via the notebooks + `scripts/seed_ml_config_tables.py`. |
